## Java was nicer than I expected

Coming from a TypeScript-heavy world — React, Node.js, Express, Prisma, NestJS — Java and Spring Boot can feel intimidating at first.

There are annotations everywhere. The project structure looks more formal. Imports matter. And some class names feel like they were generated by a committee with excellent job security.

But after building a real API with Spring Boot, something clicked:

> **Many Spring concepts are not new. They are familiar backend ideas expressed through stronger conventions.**

This article is for TypeScript and frontend engineers who want to understand Spring Boot through concepts they already know.

We’ll focus on the practical mental model:

- How Node/NestJS concepts map to Spring
- What Spring annotations actually do
- How repositories, services, validation, and exceptions fit together
- What Java features are useful for API development
- Why Spring feels verbose at first, but productive once the structure clicks

## The mental shift: Node flexibility vs Spring conventions

Node.js gives you a lot of freedom.

That freedom is great, but it also means every backend can look different:

- Express routes in one file
- Layered Express apps
- NestJS modules/providers/controllers
- Prisma used directly in services
- Custom validation and error middleware

Spring Boot is more opinionated.

A common Spring API follows this shape:

```txt
Controller -> Service -> Repository -> Database
```
![Spring API Overview](/images/blog/10-nodejs-to-springboot-ts-engineer-guide-to-java/2.png)

In plain English:

- **Controller** -> HTTP layer
- **Service** -> business logic
- **Repository** -> data access
- **DTOs** -> request/response contracts
- **Exception handler** -> consistent API errors

At first, it can feel strict. After a while, it feels productive.

You spend less time debating structure and more time building behavior.

## Concept mapping: TypeScript backend ideas -> Spring Boot ideas

![Concept mapping - typescript to spring boot](/images/blog/10-nodejs-to-springboot-ts-engineer-guide-to-java/1.png)

This mapping helped me a lot:

- **Express route handler** -> Spring `@RestController` method
- **NestJS controller** -> Spring `@RestController`
- **NestJS provider/service** -> Spring `@Service`
- **Prisma model** -> JPA `@Entity`
- **Prisma queries** -> Spring Data JPA repository
- **Zod / class-validator** -> Jakarta Bean Validation
- **Error middleware** -> `@ControllerAdvice`
- **NestJS dependency injection** -> Spring dependency injection
- **DTO/interface types** -> Java records / DTO classes
- **Transaction wrapper** -> `@Transactional`

Once this mapping clicks, Spring feels much less foreign.

### Routing: Express/NestJS controllers -> Spring controllers

In Spring Boot, controllers handle HTTP requests and delegate real work to services.

```java
@RestController
@RequestMapping("/api/booking-requests")
public class BookingRequestController {

    private final BookingRequestService bookingRequestService;

    public BookingRequestController(BookingRequestService bookingRequestService) {
        this.bookingRequestService = bookingRequestService;
    }

    @PostMapping
    public ResponseEntity<BookingRequestResponse> create(
            @Valid @RequestBody BookingRequestCreateRequest request
    ) {
        return ResponseEntity
                .status(HttpStatus.CREATED)
                .body(bookingRequestService.create(request));
    }
}
```

The important annotations:

- `@RestController` -> this class exposes JSON HTTP endpoints
- `@RequestMapping` -> base route
- `@PostMapping` -> handles POST requests
- `@RequestBody` -> maps JSON body to a Java object
- `@Valid` -> runs validation before the service is called
- `ResponseEntity` -> controls HTTP status + response body

The syntax is Java, but the flow is familiar:

```txt
Request -> Controller -> Service -> Response
```

### Validation: Zod/class-validator -> Jakarta Validation

In TypeScript projects, I often use Zod or class-validator to validate incoming payloads.

In Spring Boot, validation can live directly on DTOs.

A nice modern Java option is a `record`:

```java
public record BookingRequestCreateRequest(
        @NotNull(message = "Customer is required")
        @Valid
        CustomerRequest customer,

        @NotNull(message = "Photography package id is required")
        Long photographyPackageId,

        @NotNull(message = "Booking requested date is required")
        @Future(message = "Booking requested date must be in the future")
        OffsetDateTime requestedDate
) {}
```

Why I liked this:

- **Records reduce boilerplate** for DTOs
- Validation rules are close to the data contract
- Controllers stay clean
- Services receive already-validated input

So instead of writing manual `if (!body.email)` checks everywhere, Spring validates the request at the edge.

### Dependency injection: NestJS providers -> Spring services

If you’ve used NestJS, Spring dependency injection feels familiar.

```java
@Service
@Transactional(readOnly = true)
public class BookingRequestService {

    private final BookingRequestRepository bookingRequestRepository;

    public BookingRequestService(BookingRequestRepository bookingRequestRepository) {
        this.bookingRequestRepository = bookingRequestRepository;
    }
}
```

Key ideas:

- `@Service` -> marks this as business logic
- Constructor injection -> dependencies are explicit
- `final` -> dependency should not change
- `@Transactional` -> controls database transaction boundaries

This is one place where Spring feels very structured in a good way.

You can usually tell what a class does by its annotation and package.

### Repositories: Prisma queries -> Spring Data JPA

Prisma gives you a clean API for database access.

Spring Data JPA solves a similar problem through repositories:

```java
public interface BookingRequestRepository
        extends JpaRepository<BookingRequest, Long> {

    List<BookingRequest> findAllByOrderByCreatedAtDesc();
}
```

Yes, that method name is real.

Spring reads it and generates the query behavior.

A few examples:

```java
Optional<BookingRequest> findById(Long id);
List<BookingRequest> findByStatus(BookingStatus status);
boolean existsByCustomerEmail(String email);
```

At first, this feels like magic. Slightly suspicious magic.

But for standard queries, it is very productive.

For complex queries, you can still write explicit queries with `@Query`, so you are not locked into method-name poetry forever.

## Entities and Lombok: reducing Java boilerplate

In Prisma, your database model lives in `schema.prisma`.

In Spring/JPA, it usually lives as an entity class:

```java
@Entity
@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class BookingRequest {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Enumerated(EnumType.STRING)
    private BookingStatus status;

    private OffsetDateTime createdAt;
}
```

This is where **Lombok** helps.

Without Lombok, Java entities can get noisy with getters, setters, constructors, and builders.

With Lombok:

- `@Getter` / `@Setter` -> generated accessors
- `@NoArgsConstructor` / `@AllArgsConstructor` -> generated constructors
- `@Builder` -> builder pattern without writing it manually

The caveat: Lombok hides generated code, so it’s useful to understand what it creates before relying on it everywhere.

## Service layer: where the business rules live

The controller should not know too much.

Its job is to handle HTTP. The service handles the workflow.

```java
@Service
public class BookingRequestService {

    public BookingRequestResponse create(BookingRequestCreateRequest request) {
        BookingRequest bookingRequest = BookingRequest.builder()
                .customerEmail(request.customer().email())
                .requestedDate(request.requestedDate())
                .status(BookingStatus.PENDING)
                .createdAt(OffsetDateTime.now())
                .build();

        BookingRequest saved = bookingRequestRepository.save(bookingRequest);
        return toResponse(saved);
    }
}
```

The service layer is where you usually put:

- Business rules
- Workflow decisions
- Transactional operations
- Entity creation/update logic
- Mapping from entity -> response DTO

This is similar to a NestJS service or a well-structured Node service layer.

The big benefit is that controllers stay thin and boring.

And boring controllers are a gift to your future self.

## Java features that stood out

Java can be verbose, but modern Java has some features that make API code pleasant.

### Records

Great for immutable DTOs:

```java
public record ApiErrorResponse(
        Instant timestamp,
        int status,
        String message,
        Map<String, String> fieldErrors
) {}
```

Less boilerplate, clear intent.

### Lambdas and streams

This felt familiar coming from JavaScript:

```java
return bookingRequestRepository.findAllByOrderByCreatedAtDesc()
        .stream()
        .map(this::toResponse)
        .toList();
```

Think of it as Java’s more formal cousin of `.map()`.

### Switch expressions

Useful for workflow/state rules:

```java
return switch (currentStatus) {
    case PENDING -> newStatus == CONFIRMED || newStatus == CANCELLED;
    case CONFIRMED -> newStatus == COMPLETED || newStatus == CANCELLED;
    case CANCELLED, COMPLETED -> false;
};
```

This reads nicely when modeling business transitions.

### Optional

`Optional<T>` makes missing values explicit:

```java
BookingRequest booking = repository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException("Booking not found"));
```

It nudges you to handle the “not found” case deliberately.

## Exception handling: error middleware -> `@ControllerAdvice`

In Node/Express, you might use error middleware.

In Spring, a common pattern is global exception handling:

```java
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiErrorResponse> handleNotFound(
            ResourceNotFoundException exception
    ) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(new ApiErrorResponse(
                        Instant.now(),
                        404,
                        exception.getMessage(),
                        Map.of()
                ));
    }
}
```

This gives your API a consistent error shape.

That is especially useful for the frontend because it can reliably read:

- `message`
- `status`
- `fieldErrors`

Instead of guessing how each endpoint formats errors.

## Spring annotations worth knowing early

Spring is annotation-heavy, but the annotations usually express architecture.

High ROI annotations:

- `@SpringBootApplication` -> app entry point + auto-configuration
- `@RestController` -> JSON HTTP controller
- `@RequestMapping` -> base route
- `@GetMapping`, `@PostMapping`, etc. -> route handlers
- `@RequestBody` -> JSON body binding
- `@PathVariable` -> URL path value
- `@RequestParam` -> query string value
- `@Valid` -> validation trigger
- `@Service` -> business layer
- `@Repository` -> persistence layer marker
- `@Transactional` -> transaction boundary
- `@ControllerAdvice` -> global exception handling
- `@ExceptionHandler` -> handle a specific exception
- `@Configuration` -> custom app configuration

At first, annotations feel like magic.

After a while, they feel like a declarative language for wiring an API.

## Frontend integration: keeping it simple

![APP Overview](/images/blog/10-nodejs-to-springboot-ts-engineer-guide-to-java/3.png)

The frontend part was intentionally small:

- Vite + React
- API client using `VITE_API_BASE_URL`
- Forms calling the Spring API
- Error responses mapped to UI messages

The main backend integration piece was **CORS**.

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("http://localhost:5173")
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*");
    }
}
```

The main lesson:

> **When the backend has clear DTOs and consistent error responses, frontend integration becomes much easier.**

## What felt better than expected

A few honest takeaways:

- **Java is nicer than I expected**, especially with records, lambdas, and streams
- **Spring conventions reduce decision fatigue**
- **Validation and exception handling feel mature**
- **Repositories make standard database queries quick to write**
- **Layered architecture makes onboarding easier**

For API development, the structure is a feature.

## What still felt annoying

Of course, not everything was magical.

- **Imports matter a lot** — wrong imports can waste time
- **Java can still be verbose**, even with Lombok
- **Annotation-heavy code takes practice to read**
- **Spring magic is useful**, but you should understand what it’s doing

The magic is productive until something breaks. Then you want to know where the wires are.

## When I would reach for Spring Boot

Spring Boot makes a lot of sense for:

- CRUD APIs
- Internal business systems
- APIs with validation-heavy workflows
- Services that need database transactions
- Teams that benefit from conventions
- Projects where reliability and maintainability matter

Would I use it for every tiny prototype? Probably not.

Would I use it for a serious API? Yes.

## Key takeaways for TypeScript engineers

If you’re coming from Node.js, Express, Prisma, or NestJS, Spring Boot is less alien than it first appears.

The vocabulary changes, but the ideas are familiar:

- **Controllers** handle HTTP
- **Services** handle business logic
- **Repositories** handle data access
- **DTOs** define API boundaries
- **Validation** happens at the edge
- **Exceptions** are centralized
- **Transactions** are explicit

For me, the biggest realization was this:

> **Java/Spring trades some flexibility and terseness for consistency, safety, and predictability — and for API work, that trade often makes a lot of sense.**

You may still fight imports occasionally.

You may stare at `findAllByOrderByCreatedAtDesc()` and wonder if Java has finally gone too far.

But overall?

Java is rather nice.

Spring Boot is solid.

And if you already understand backend concepts from Node.js, you are closer than you think.

## Deploying it — Render, Vercel and Postgres

To round off the project, I deployed the backend and frontend separately:

- **Spring Boot API** -> Render
- **PostgreSQL database** -> Render Postgres for the deployed app
- **React frontend** -> Vercel
- **Docker** -> used locally for running PostgreSQL during development

This separation felt natural:

```txt
React frontend on Vercel
        -> calls
Dockerized Spring Boot API on Render
        -> connects to
PostgreSQL on Render
```

### Render: Java + Spring Boot + PostgreSQL

The backend is prepared for a **Docker-based deployment on Render**. This keeps the Java runtime predictable, which is useful because the project uses **Java 26**. Instead of relying on whatever Java version the platform provides, the Dockerfile defines the runtime environment explicitly.

The deployment setup includes:

- `Dockerfile` and `.dockerignore` for packaging the Spring Boot API
- `render.yaml` for the Render service configuration
- GitHub Actions workflows for CI and deployment
- Render PostgreSQL for the production database

At a high level, the CI/CD flow is:

```txt
Pull request / push to main
        -> run tests
        -> verify Docker build
        -> deploy to Render after CI succeeds
```

Render automatic deploys are disabled, so GitHub Actions can control production deployment through a Render deploy hook.

A small caveat: if you use Render’s free tier, the backend may sleep after inactivity, so the first request can be slower.

### Vercel: React + Vite

For the frontend, Vercel was straightforward:

- Add the frontend project
- Set the API base URL environment variable
- Deploy from GitHub
- Let Vercel handle previews and production builds

---
The important lesson here was simple:

> **A full-stack app is not complete until the deployment story works too.**

Local development is one thing. But once the frontend, backend, database, environment variables, CORS, and deployment platforms all have to talk to each other, you start seeing the project as a real system.
